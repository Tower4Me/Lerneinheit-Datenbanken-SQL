Modul 01 · Datenbanken

Was ist eine Datenbank?

Eine Datenbank ist ein System zur strukturierten, dauerhaften Speicherung von Daten, auf die mehrere Programme oder Benutzer gleichzeitig zugreifen können.

Stell dir vor, du hast 10.000 Kundendaten in einer Excel-Tabelle. Das funktioniert für kleine Mengen. Aber was passiert, wenn zwei Mitarbeiter gleichzeitig einen Eintrag ändern? Was wenn du gezielt alle Kunden aus NRW brauchst, die in den letzten 30 Tagen bestellt haben? Excel gerät schnell an seine Grenzen.

Genau daفür gibt es Datenbanken und das Programm das sie verwaltet: das Datenbankmanagementsystem (DBMS).

Abkürzung

DBMS = Datenbankmanagementsystem (engl. Database Management System). Das ist die Software, die zwischen dir und den eigentlichen Daten sitzt. Beispiele: MySQL, PostgreSQL, SQLite, Microsoft SQL Server.

Persistent vs. flüchtig

Persistent

Dauerhaft gespeichert

Daten bleiben erhalten, auch wenn der Strom weg ist. Datenbanken speichern auf Festplatte (HDD/SSD).

Flüchtig

Nur im Arbeitsspeicher

Der RAM (Random Access Memory) verliert alle Daten bei Stromunterbrechung. Nur für temporäre Berechnungen.

Was ist relational?

Statt alles in einer riesigen Tabelle zu speichern, verteilst du Daten auf mehrere Tabellen und verknüpfst sie miteinander. Jede Abteilung bekommt eine ID, der Mitarbeiter speichert nur diese ID – kein doppelter Abteilungsname bei jedem Mitarbeiter.

■ mitarbeiter
idINTPK
nameVARCHAR
abteilung_idINTFK
■ abteilung
idINTPK
bezeichnungVARCHAR
standortVARCHAR
PK / FK erklärt

PK (Primary Key = Primärschlüssel): Eindeutige ID jedes Datensatzes. Kein zweiter Datensatz darf denselben PK haben.

FK (Foreign Key = Fremdschlüssel): Verweist auf den PK einer anderen Tabelle. So entsteht die Verbindung.

Das ACID-Prinzip

Ein gutes DBMS garantiert vier Eigenschaften. Das ist der Grund warum Datenbanken zuverlässiger sind als einfache Dateien.

A · Atomicity

Atomarität

Eine Transaktion wird entweder vollständig ausgeführt oder gar nicht. Kein halbfertiger Zustand möglich.

C · Consistency

Konsistenz

Die Datenbank bleibt immer in einem gültigen Zustand. Regeln und Constraints werden nie verletzt.

I · Isolation

Isolation

Gleichzeitige Transaktionen stören sich nicht gegenseitig. Jede läuft so, als wäre sie die einzige.

D · Durability

Dauerhaftigkeit

Abgeschlossene Transaktionen bleiben gespeichert, auch bei Systemabsturz.

Merkhilfe ACID

Banküberweisung: Entweder läuft sie komplett durch (Atomarität), das Konto bleibt nicht im Minus (Konsistenz), andere Überweisungen warten (Isolation), und die Buchung bleibt nach Absturz erhalten (Dauerhaftigkeit).

Verbindung zu Modul 8

ACID setzt voraus, dass du Transaktionen steuern kannst. Das lernst du in Modul 8 (BEGIN, COMMIT, ROLLBACK). ACID ist das Warum – Transaktionen sind das Wie.

Modul 01 · Datenbanken

Arten von Datenbanken

Nicht jedes Problem braucht eine relationale Datenbank. Es gibt verschiedene Typen für unterschiedliche Anwendungsfälle.

Relationale Datenbank (RDBMS)

Daten in Tabellen mit festen Spalten, verknüpft über Schlüssel. Abfragen laufen über SQL.

Stärken
  • Klare Struktur, einfach zu verstehen
  • ACID-Garantien, sehr zuverlässig
  • SQL ist weit verbreitet und standardisiert
  • Gut für komplexe Abfragen über mehrere Tabellen
Schwächen
  • Festes Schema muss vorher definiert werden
  • Schlecht skalierbar bei Milliarden Datensätzen
  • Unflexibel bei häufig wechselnder Datenstruktur

Beispiele: MySQL, PostgreSQL, SQLite, Microsoft SQL Server, Oracle DB
Einsatz: Shopsysteme, ERP-Software, Buchhaltung

Dokumentenbasierte Datenbank (NoSQL)

Kein festes Schema. Daten als JSON-ähnliche Dokumente. Jedes Dokument kann eine andere Struktur haben.

Stärken
  • Sehr flexibel, kein festes Schema nötig
  • Gut skalierbar für riesige Datenmengen
  • Schnell bei einfachen Lesezugriffen
Schwächen
  • Keine ACID-Garantien (je nach System)
  • Komplexe Abfragen über Beziehungen schwieriger
  • Kein einheitlicher Standard wie SQL

Beispiele: MongoDB, CouchDB, Firebase
Einsatz: Social Media, Apps mit variierenden Datenstrukturen

Key-Value Store

Einfachste Form: Schlüssel zeigt auf Wert. Wie ein Schrank mit nummerierten Fächern.

Stärken
  • Extrem schnell (oft im RAM)
  • Ideal für temporäre Daten (Sessions, Cache)
Schwächen
  • Keine Abfragen über Inhalte möglich
  • Keine Beziehungen zwischen Daten

Beispiele: Redis, Memcached
Einsatz: Login-Sessions, Caching

Wann nimmt man was?

SituationEmpfehlungWarum
Klassisches Unternehmen, ERP, ShopRelational (MySQL/PostgreSQL)Strukturiert, ACID, SQL bekannt
App mit variablen DatenformatenDokumentenbasiert (MongoDB)Kein festes Schema nötig
Login-Sessions, schneller CacheKey-Value (Redis)Schnell, einfach, temporär
Kleines Projekt, lokale AppSQLiteKeine Installation, eine Datei
Für die IHK

In der Prüfung dreht sich fast alles um relationale Datenbanken und SQL. NoSQL und Key-Value werden höchstens im Überblick abgefragt.

Modul 01 · Datenbanken

Aufbau & Grundbegriffe

Bevor du SQL schreibst, musst du verstehen wie eine relationale Datenbank von innen aufgebaut ist.

Die Hierarchie

1. Server / Instanz

Der DBMS-Server

Das laufende Programm (z.B. MySQL-Server). Auf einem Server können viele Datenbanken laufen.

2. Datenbank

Die Datenbank

Ein benannter Container mit zusammengehörigen Tabellen. Z.B. eine Datenbank "firma_gmbh".

3. Schema

Das Schema

Die Struktur: welche Tabellen gibt es, welche Spalten, welche Datentypen. In vielen Systemen = Datenbank.

4. Tabelle

Die Tabelle

Enthält Daten in Zeilen (Datensätze) und Spalten (Felder). Wie ein Excel-Tabellenblatt, aber strenger strukturiert.

Modul 02 · SQL Grundlagen

Was ist SQL?

SQL (Structured Query Language = Strukturierte Abfragesprache) ist die Standardsprache für relationale Datenbanken. Du nutzt sie um Daten abzufragen, zu verändern, Strukturen zu definieren und Rechte zu vergeben.

Die drei Unterbereiche

DDL

Data Definition Language

Datenbankstruktur verändern. Tabellen erstellen, ändern, löschen.
Befehle: CREATE, ALTER, DROP, TRUNCATE

DML

Data Manipulation Language

Daten lesen und verändern.
Befehle: SELECT, INSERT, UPDATE, DELETE

DCL

Data Control Language

Zugriffsrechte steuern.
Befehle: GRANT, REVOKE

Merkhilfe

DDL = Regale bauen. DML = Bücher rein- und raustragen. DCL = Schlüssel ausgeben oder einziehen.

Was kann SQL?

AufgabeBereichBeispielbefehl
Tabelle erstellenDDLCREATE TABLE
Daten abfragenDMLSELECT
Daten einfügenDMLINSERT INTO
Daten ändernDMLUPDATE
Daten löschenDMLDELETE FROM
Rechte vergebenDCLGRANT

Wie sieht SQL aus?

SQL-- Alle Mitarbeiter aus der IT-Abteilung, sortiert nach Name SELECT name, gehalt FROM mitarbeiter WHERE abt_id = 3 ORDER BY name ASC;
Was ist eine Query?

Eine Query (Abfrage) ist ein SQL-Befehl der eine Anfrage an die Datenbank stellt. Jede Query endet mit einem Semikolon ;. SQL ist nicht case-sensitiv: select und SELECT sind identisch. Konvention: Schlüsselwörter in Großbuchstaben.

Was ist ein Schema?

Das Schema ist der Bauplan der Datenbank: welche Tabellen, welche Spalten, welche Datentypen, welche Beziehungen. Die eigentlichen Daten sind der Inhalt. Schema = Struktur, Daten = Inhalt.

Modul 03 · Datentypen

Datentypen & Constraints

Jede Spalte in einer Tabelle hat einen Datentyp. Der Datentyp legt fest was dort gespeichert werden darf und wie viel Speicher es braucht.

Numerische Datentypen

TypBeschreibungBeispiel
INT / INTEGERGanze Zahl (–2 Mrd. bis +2 Mrd.)42, –7, 1000
BIGINTSehr große ganze Zahl9.223.372.036.854.775.807
DECIMAL(p,s)Exakte Dezimalzahl, p Stellen gesamt, s NachkommaDECIMAL(10,2) = 99999999.99
FLOAT / DOUBLEGleitkommazahl (Rundungsfehler möglich!)3.14159...
Wichtig: Geldbeträge

Für Preise und Geldbeträge immer DECIMAL(10,2) verwenden, niemals FLOAT oder DOUBLE. Grund: 0.1 + 0.2 ergibt in Binärdarstellung nicht exakt 0.3. Mit FLOAT würde der Lohnzettel falsch sein.

Text-Datentypen

TypBeschreibungEinsatz
CHAR(n)Feste Länge n Zeichen (füllt mit Leerzeichen auf)ISO-Codes ("DE", "US")
VARCHAR(n)Variable Länge, max n ZeichenNamen, E-Mails, Beschreibungen
TEXTGroßer Text, unbegrenztKommentare, Artikel, Beschreibungen

Datum & Zeit

TypFormatBeispiel
DATEYYYY-MM-DD2024-03-15
TIMEHH:MM:SS14:30:00
DATETIME / TIMESTAMPYYYY-MM-DD HH:MM:SS2024-03-15 14:30:00

Binäre Datentypen (BLOB)

BLOB (Binary Large Object) = Binäres großes Objekt. Für rohe Binärdaten: Bilder, Dateien, Audio.

TypMax. Größe
TINYBLOB255 Byte
BLOB65 KB
MEDIUMBLOB16 MB
LONGBLOB4 GB

Constraints (Einschränkungen)

Constraints sind Regeln die du auf Spalten anwendest. Sie schützen die Datenintegrität.

PRIMARY KEY

Primärschlüssel

Eindeutig + NOT NULL. Jede Tabelle sollte einen haben. Identifiziert jeden Datensatz eindeutig.

NOT NULL

Pflichtfeld

Das Feld muss einen Wert haben. NULL ist nicht erlaubt.

UNIQUE

Eindeutig

Kein zweiter Datensatz darf denselben Wert in dieser Spalte haben. Z.B. für E-Mail-Adressen.

DEFAULT

Standardwert

Wenn beim Einfügen kein Wert angegeben wird, nimmt das Feld automatisch diesen Wert an.

CHECK

Bedingung

Der Wert muss eine Bedingung erfüllen. Z.B. Alter muss größer als 0 sein.

FOREIGN KEY

Fremdschlüssel

Verweist auf den PK einer anderen Tabelle. Stellt sicher, dass keine verwaisten Verweise entstehen.

Constraints in der Praxis

SQLCREATE TABLE mitarbeiter ( id INT PRIMARY KEY AUTO_INCREMENT, -- MySQL name VARCHAR(100) NOT NULL, email VARCHAR(150) NOT NULL UNIQUE, gehalt DECIMAL(10,2) DEFAULT 0.00, alter_j INT CHECK (alter_j > 0 AND alter_j < 120), abt_id INT, FOREIGN KEY (abt_id) REFERENCES abteilung(id) );
Dialekt-Hinweis: CHECK in MySQL

CHECK-Constraints wurden in MySQL erst ab Version 8.0.16 wirklich durchgesetzt. In älteren Versionen wurden sie akzeptiert aber ignoriert. In PostgreSQL und SQLite funktionieren sie zuverlässig.

Modul 04 · DDL & Normalisierung

DDL – Data Definition Language

DDL (Data Definition Language = Datendefinitionssprache) umfasst alle Befehle, die die Struktur einer Datenbank verändern. Nicht die Daten selbst, sondern das Gerüst drumherum.

Merkhilfe

DDL = Regale bauen. Du legst fest wie die Datenbank aussieht. Erst wenn das Regal steht, kannst du Bücher (Daten) reinstellen – das ist dann DML.

CREATE TABLE

Erstellt eine neue Tabelle. Erst danach können Daten gespeichert werden.

SQLiteCREATE TABLE abteilung ( id INTEGER PRIMARY KEY AUTOINCREMENT, bezeichnung VARCHAR(100) NOT NULL, standort VARCHAR(100) );
Reihenfolge wichtig

Wenn Tabelle A einen Fremdschlüssel auf Tabelle B hat, muss Tabelle B zuerst erstellt werden. abteilung vor mitarbeiter, weil mitarbeiter einen FK auf abteilung hat.

ALTER TABLE

Ändert eine bestehende Tabelle. Spalten hinzufügen, umbenennen, entfernen.

SQLALTER TABLE mitarbeiter ADD COLUMN einstellungsdatum DATE; ALTER TABLE mitarbeiter RENAME COLUMN email TO email_adresse; ALTER TABLE mitarbeiter RENAME TO personal;

DROP TABLE

Löscht eine Tabelle vollständig – inklusive aller Daten. Nicht rückgängig machbar.

SQLDROP TABLE mitarbeiter; DROP TABLE IF EXISTS mitarbeiter;
Achtung

DROP TABLE löscht die Tabelle und alle Daten darin permanent. Kein Undo. Immer vorher Backup machen.

TRUNCATE TABLE

Leert eine Tabelle vollständig, behält aber die Tabellenstruktur.

MySQL / PostgreSQLTRUNCATE TABLE mitarbeiter;
Dialekt-Hinweis: SQLite

SQLite kennt kein TRUNCATE. Stattdessen: DELETE FROM mitarbeiter;

Übersicht DDL-Befehle

BefehlWirkungDaten weg?Struktur weg?
CREATE TABLENeue Tabelle anlegen––
ALTER TABLETabelle ändernNeinNein
DROP TABLETabelle löschenJaJa
TRUNCATE TABLEAlle Daten leerenJaNein

Index

Ein Index beschleunigt Abfragen auf bestimmte Spalten. Ohne Index: Full Table Scan. Mit Index: direkte Suche.

SQLCREATE INDEX idx_email ON mitarbeiter(email); DROP INDEX idx_email;
Vorteile
  • SELECT-Abfragen auf indizierte Spalten viel schneller
  • PK und UNIQUE sind automatisch indiziert
Nachteile
  • Zusätzlicher Speicherplatz
  • INSERT/UPDATE/DELETE langsamer (Index muss aktualisiert werden)
Modul 04 · DDL & Normalisierung

Normalisierung

Normalisierung ist der Prozess, eine Datenbank so zu strukturieren, dass Redundanzen (doppelte Daten) und Anomalien vermieden werden. Die IHK erwartet, dass du eine Tabelle schrittweise normalisieren kannst.

Warum normalisieren?

Ohne Normalisierung entsteht dieses Problem: Wenn eine Abteilung umzieht, müsstest du den Standort bei jedem Mitarbeiter einzeln ändern. Vergisst du einen, sind die Daten inkonsistent. Mit Normalisierung änderst du nur eine Zeile in der Abteilungs-Tabelle.

Ausgangslage: Unnormalisierte Tabelle (0NF)

Stellen wir uns diese Tabelle vor:

bestellung_idkunde_namekunde_stadtplzartikelpreis
1Anna MüllerKöln50667Laptop, Maus1200, 25
2Ben SchmidtBerlin10115Monitor350
3Anna MüllerKöln50667Tastatur80

Probleme: Anna Müllers Stadt steht doppelt. Die Spalten "artikel" und "preis" enthalten Listen (mehrere Werte in einer Zelle). PLZ bestimmt die Stadt – das ist eine transitive Abhängigkeit.

1NF: Atomare Werte

Regel: Jede Zelle enthält genau einen einzelnen Wert. Keine Listen, keine Mehrfachwerte.

bestellung_idkunde_namekunde_stadtplzartikelpreis
1Anna MüllerKöln50667Laptop1200
1Anna MüllerKöln50667Maus25
2Ben SchmidtBerlin10115Monitor350
3Anna MüllerKöln50667Tastatur80
1NF erreicht

Jede Zelle hat jetzt genau einen Wert. Das Problem: Annas Daten wiederholen sich noch. Weiter zu 2NF.

2NF: Keine Teilabhängigkeit

Regel: Jede Nicht-Schlüsselspalte hängt vom gesamten Primärschlüssel ab, nicht nur von einem Teil davon. Gilt nur bei zusammengesetzten PKs.

In unserer Tabelle ist der PK zusammengesetzt: (bestellung_id, artikel). Aber kunde_name und plz hängen nur von bestellung_id ab, nicht von artikel. Das ist eine Teilabhängigkeit → Lösung: aufteilen.

Tabelle: bestellung
  • bestellung_id (PK)
  • kunde_name
  • kunde_stadt
  • plz
Tabelle: bestellung_position
  • bestellung_id (FK)
  • artikel
  • preis
2NF erreicht

Artikeldaten und Kundendaten sind getrennt. Aber: plz bestimmt noch die Stadt. Das ist eine transitive Abhängigkeit → 3NF.

3NF: Keine transitiven Abhängigkeiten

Regel: Kein Nicht-Schlüsselfeld darf von einem anderen Nicht-Schlüsselfeld abhängen.

Problem: plz → kunde_stadt. Die Stadt wird durch die PLZ bestimmt, nicht durch die bestellung_id. Das muss ausgelagert werden.

Tabelle: bestellung
  • bestellung_id (PK)
  • kunde_name
  • plz (FK)
Tabelle: plz_ort
  • plz (PK)
  • stadt
3NF erreicht

Jetzt ist alles sauber. Stadt steht nur noch in der PLZ-Tabelle. Ändert sich der Stadtname für eine PLZ, änderst du eine Zeile in plz_ort – fertig.

Für die IHK

Die IHK kann eine unnormalisierte Tabelle geben und verlangen, dass du 1NF, 2NF und 3NF Schritt für Schritt anwendest. Lerne die drei Regeln auswendig und übe an Beispielen.

Modul 05 · DML

DML – Data Manipulation Language

DML (Data Manipulation Language = Datenmanipulationssprache) umfasst alle Befehle zum Lesen und Verändern von Daten. Das Gerüst steht (DDL), jetzt kommen die Daten rein.

SELECT – Daten lesen

SQL-- Alle Spalten SELECT * FROM mitarbeiter; -- Bestimmte Spalten SELECT name, gehalt FROM mitarbeiter; -- Mit Alias SELECT name AS mitarbeitername, gehalt AS monatslohn FROM mitarbeiter;

WHERE – Bedingungen

SQLSELECT * FROM mitarbeiter WHERE abt_id = 3; SELECT * FROM mitarbeiter WHERE gehalt > 3000 AND abt_id = 2; SELECT * FROM mitarbeiter WHERE name LIKE 'A%'; -- beginnt mit A SELECT * FROM mitarbeiter WHERE gehalt BETWEEN 2000 AND 4000; SELECT * FROM mitarbeiter WHERE abt_id IN (1, 3, 5); SELECT * FROM mitarbeiter WHERE email IS NULL;

ORDER BY & LIMIT

SQL-- Sortierung: ASC (aufsteigend, Standard), DESC (absteigend) SELECT name, gehalt FROM mitarbeiter ORDER BY gehalt DESC; -- Nur die ersten 5 Ergebnisse SELECT * FROM mitarbeiter ORDER BY name ASC LIMIT 5; -- Die 3 am besten verdienenden Mitarbeiter SELECT name, gehalt FROM mitarbeiter ORDER BY gehalt DESC LIMIT 3;
Für die IHK

ORDER BY und LIMIT tauchen oft in Prüfungsaufgaben auf: "Zeige die 5 teuersten Produkte" oder "sortiere nach Name". Lerne die Syntax auswendig.

INSERT – Daten einfügen

SQL-- Mit Spaltennamen (empfohlen) INSERT INTO mitarbeiter (name, email, gehalt, abt_id) VALUES ('Anna Müller', 'anna@firma.de', 3200.00, 1); -- Mehrere Zeilen auf einmal INSERT INTO abteilung (bezeichnung, standort) VALUES ('IT', 'Köln'), ('HR', 'Berlin'), ('Vertrieb', 'München');

UPDATE – Daten ändern

SQLUPDATE mitarbeiter SET gehalt = 3500.00 WHERE id = 1;
Gefährlich: UPDATE ohne WHERE

UPDATE mitarbeiter SET gehalt = 0; – das setzt das Gehalt ALLER Mitarbeiter auf 0. Immer erst mit SELECT prüfen welche Zeilen betroffen sind.

DELETE – Daten löschen

SQL-- Erst prüfen: SELECT * FROM mitarbeiter WHERE id = 5; -- Dann löschen: DELETE FROM mitarbeiter WHERE id = 5;

Aggregat-Funktionen

SQLSELECT COUNT(*) FROM mitarbeiter; -- Anzahl aller Zeilen SELECT COUNT(email) FROM mitarbeiter; -- Anzahl ohne NULL SELECT SUM(gehalt) FROM mitarbeiter; -- Summe SELECT AVG(gehalt) FROM mitarbeiter; -- Durchschnitt SELECT MAX(gehalt) FROM mitarbeiter; -- Maximum SELECT MIN(gehalt) FROM mitarbeiter; -- Minimum

GROUP BY & HAVING

SQL-- Durchschnittsgehalt je Abteilung SELECT abt_id, AVG(gehalt) AS durchschnitt FROM mitarbeiter GROUP BY abt_id; -- Nur Abteilungen mit Durchschnitt über 3000 SELECT abt_id, AVG(gehalt) AS durchschnitt FROM mitarbeiter GROUP BY abt_id HAVING AVG(gehalt) > 3000;
WHERE vs. HAVING

WHERE filtert einzelne Zeilen vor der Gruppierung. Kann keine Aggregat-Funktionen verwenden.
HAVING filtert Gruppen nach GROUP BY. Kann AVG(), COUNT() etc. verwenden.

Modul 06 · JOIN

JOIN – Tabellen verknüpfen

JOIN kombiniert Zeilen aus zwei oder mehr Tabellen auf Basis einer gemeinsamen Spalte (meist PK/FK). Ohne JOIN müsstest du mehrere Abfragen machen und das Ergebnis selbst zusammenbauen.

Warum JOIN?

Mitarbeiter haben eine abt_id, aber keine Abteilungsbezeichnung. Mit JOIN holst du dir die Bezeichnung direkt aus der Abteilungs-Tabelle.

SQL – Ohne JOIN-- Erst Abteilung 3 suchen: SELECT id FROM abteilung WHERE bezeichnung = 'IT'; -- Dann Mitarbeiter dieser Abteilung: SELECT * FROM mitarbeiter WHERE abt_id = 3; -- Mit JOIN in einer Abfrage: SELECT m.name, a.bezeichnung FROM mitarbeiter m INNER JOIN abteilung a ON m.abt_id = a.id;

INNER JOIN

Gibt nur Zeilen zurück, die in beiden Tabellen eine Übereinstimmung haben. Mitarbeiter ohne Abteilung und Abteilungen ohne Mitarbeiter werden nicht angezeigt.

SQLSELECT m.name, m.gehalt, a.bezeichnung AS abteilung FROM mitarbeiter m INNER JOIN abteilung a ON m.abt_id = a.id;

LEFT JOIN

Gibt alle Zeilen der linken Tabelle zurück, auch wenn es keine Übereinstimmung in der rechten gibt. Fehlende Werte werden als NULL angezeigt.

SQL-- Alle Mitarbeiter, auch ohne Abteilung SELECT m.name, a.bezeichnung FROM mitarbeiter m LEFT JOIN abteilung a ON m.abt_id = a.id;

RIGHT JOIN

Gibt alle Zeilen der rechten Tabelle zurück, auch ohne Übereinstimmung links. In SQLite nicht verfügbar (Workaround: Tabellen tauschen + LEFT JOIN).

SQL (MySQL/PostgreSQL)-- Alle Abteilungen, auch solche ohne Mitarbeiter SELECT m.name, a.bezeichnung FROM mitarbeiter m RIGHT JOIN abteilung a ON m.abt_id = a.id;

FULL OUTER JOIN

Gibt alle Zeilen aus beiden Tabellen zurück, unabhängig von Übereinstimmungen. Nicht in MySQL direkt verfügbar (Workaround: LEFT JOIN UNION RIGHT JOIN).

SQL (PostgreSQL / Standard)SELECT m.name, a.bezeichnung FROM mitarbeiter m FULL OUTER JOIN abteilung a ON m.abt_id = a.id;

Vergleich der JOIN-Typen

JOIN-TypZeigtTypischer Einsatz
INNER JOINNur Übereinstimmungen in beiden TabellenMitarbeiter mit Abteilung
LEFT JOINAlle links, passende rechts (oder NULL)Alle Mitarbeiter, auch ohne Abteilung
RIGHT JOINAlle rechts, passende links (oder NULL)Alle Abteilungen, auch leere
FULL OUTER JOINAlle Zeilen aus beiden SeitenVollständige Übersicht beider Tabellen

JOIN mit mehreren Tabellen

SQLSELECT m.name, a.bezeichnung, s.stadt FROM mitarbeiter m INNER JOIN abteilung a ON m.abt_id = a.id INNER JOIN standort s ON a.standort_id = s.id;

Modul 07 · Subqueries

Subqueries – Unterabfragen

Eine Subquery (Unterabfrage) ist eine SELECT-Abfrage innerhalb einer anderen Abfrage. Sie löst Probleme, bei denen du das Ergebnis einer Abfrage als Eingabe für eine andere benötigst.

Wann braucht man Subqueries?

Angenommen du willst alle Mitarbeiter anzeigen, die mehr als der Durchschnitt verdienen. Du weisst den Durchschnitt nicht – den musst du erst berechnen. Mit Subquery geht das in einer Abfrage.

SQL – Ohne Subquery (2 Schritte)-- Schritt 1: Durchschnitt berechnen SELECT AVG(gehalt) FROM mitarbeiter; -- Ergebnis: z.B. 3200 -- Schritt 2: Mitarbeiter mit Gehalt über 3200 SELECT name, gehalt FROM mitarbeiter WHERE gehalt > 3200;
SQL – Mit Subquery (1 Schritt)SELECT name, gehalt FROM mitarbeiter WHERE gehalt > (SELECT AVG(gehalt) FROM mitarbeiter);
Wie es funktioniert

Die innere Abfrage (SELECT AVG(gehalt) FROM mitarbeiter) wird zuerst ausgeführt und liefert eine Zahl. Diese Zahl wird dann in der äußeren WHERE-Bedingung eingesetzt.

Subquery im WHERE mit IN

Häufig werden Subqueries mit IN kombiniert, wenn die innere Abfrage mehrere Werte liefert.

SQL-- Alle Mitarbeiter in Abteilungen die in Köln sind SELECT name FROM mitarbeiter WHERE abt_id IN ( SELECT id FROM abteilung WHERE standort = 'Köln' );

Subquery im FROM (abgeleitete Tabelle)

Eine Subquery kann auch im FROM stehen und wird dann wie eine temporäre Tabelle behandelt. Sie braucht einen Alias.

SQL-- Durchschnittsgehalt je Abteilung, nur Abteilungen mit > 3 Mitarbeitern SELECT abt_id, durchschnitt FROM ( SELECT abt_id, AVG(gehalt) AS durchschnitt, COUNT(*) AS anzahl FROM mitarbeiter GROUP BY abt_id ) AS abt_stats WHERE anzahl > 3;

EXISTS – Existenzprüfung

EXISTS prüft ob eine Subquery mindestens eine Zeile zurückgibt. Nützlich wenn du nur wissen willst ob etwas existiert, nicht was.

SQL-- Abteilungen, in denen mindestens ein Mitarbeiter ist SELECT bezeichnung FROM abteilung a WHERE EXISTS ( SELECT 1 FROM mitarbeiter m WHERE m.abt_id = a.id );
Für die IHK

Subqueries tauchen in IHK-Aufgaben auf: "Zeige alle Produkte die teurer als der Durchschnitt sind" oder "Welche Kunden haben noch keine Bestellung aufgegeben". Die häufigsten Formen sind: Subquery in WHERE mit Vergleichsoperator, Subquery mit IN, und EXISTS.

Modul 08 · Transaktionen

Transaktionen

Eine Transaktion ist eine Gruppe von SQL-Befehlen, die zusammen als eine Einheit behandelt werden. Entweder werden alle ausgeführt (COMMIT) oder keiner (ROLLBACK). Das ist die praktische Umsetzung des ACID-Prinzips aus Modul 1.

Warum Transaktionen?

Stell dir eine Banküberweisung vor: Geld geht von Konto A ab und kommt auf Konto B an. Was passiert wenn nach dem Abbuchen vom Konto A der Server abstürzt, bevor der Gutschrift auf Konto B ausgeführt wird? Ohne Transaktion ist das Geld weg. Mit Transaktion wird der gesamte Vorgang rückgängig gemacht.

SQL – Ohne Transaktion (gefährlich)UPDATE konto SET guthaben = guthaben - 500 WHERE id = 1; -- << Server-Absturz hier >> -- Geld weg, Gegenbuchung fehlt! UPDATE konto SET guthaben = guthaben + 500 WHERE id = 2;
SQL – Mit Transaktion (sicher)BEGIN; -- oder: START TRANSACTION; (MySQL) UPDATE konto SET guthaben = guthaben - 500 WHERE id = 1; UPDATE konto SET guthaben = guthaben + 500 WHERE id = 2; COMMIT; -- Beide Buchungen dauerhaft speichern

BEGIN, COMMIT, ROLLBACK

BEGIN

Transaktion starten

Markiert den Startpunkt. Alle folgenden Befehle gehören zur Transaktion. MySQL: START TRANSACTION oder BEGIN.

COMMIT

Dauerhaft speichern

Macht alle Änderungen der Transaktion permanent. Danach gibt es kein Zurück mehr.

ROLLBACK

Alles rückgängig

Macht alle Änderungen seit BEGIN rückgängig. Als hätte die Transaktion nie stattgefunden.

Typisches Muster mit Fehlerbehandlung

SQL (Pseudocode/PostgreSQL)BEGIN; UPDATE lager SET bestand = bestand - 1 WHERE artikel_id = 42; INSERT INTO bestellung_position (bestellung_id, artikel_id, menge) VALUES (100, 42, 1); -- Wenn alles gut: speichern COMMIT; -- Wenn ein Fehler passiert: rückgängig machen -- ROLLBACK;

SAVEPOINT

Ein SAVEPOINT ist ein Zwischenpunkt innerhalb einer Transaktion. Man kann bis zu einem bestimmten Savepoint zurückrollen ohne die gesamte Transaktion abzubrechen.

SQLBEGIN; INSERT INTO bestellung ...; SAVEPOINT nach_bestellung; INSERT INTO bestellung_position ...; -- Nur die Position rückgängig, Bestellung bleibt: ROLLBACK TO nach_bestellung; COMMIT;

Autocommit

In den meisten DBMS läuft jeder einzelne SQL-Befehl automatisch in einer eigenen Transaktion (Autocommit = ON). Das bedeutet: jedes INSERT, UPDATE, DELETE wird sofort dauerhaft gespeichert. Mit BEGIN schaltest du Autocommit für diesen Block ab.

Für die IHK

Du musst erklären können was BEGIN, COMMIT und ROLLBACK machen und warum Transaktionen das ACID-Prinzip umsetzen. Das Bankbeispiel ist der Klassiker.

Modul 09 · DCL & Rollen

DCL – Data Control Language

DCL (Data Control Language = Datenkontrollsprache) steuert wer was in der Datenbank darf. Mit GRANT vergibst du Rechte, mit REVOKE entziehst du sie.

Warum Rechteverwaltung?

In einem Unternehmen soll der Azubi nicht alle Tabellen löschen dürfen. Die Buchhaltung soll keine Personaldaten sehen. Ein Webshop-Benutzer braucht kein DROP TABLE. Rechteverwaltung ist ein Kernaspekt der Datensicherheit.

GRANT – Rechte vergeben

SQL-- Einzelnes Recht GRANT SELECT ON mitarbeiter TO 'anna'; -- Mehrere Rechte auf einmal GRANT SELECT, INSERT ON mitarbeiter TO 'azubi_benutzer'; -- Alle Rechte auf eine Tabelle GRANT ALL PRIVILEGES ON mitarbeiter TO 'admin'; -- Alle Rechte auf alle Tabellen GRANT ALL PRIVILEGES ON *.* TO 'super_admin';

WITH GRANT OPTION

Wenn du einem Benutzer ein Recht mit WITH GRANT OPTION gibst, darf dieser Benutzer das Recht seinerseits an andere weitergeben.

SQLGRANT SELECT ON mitarbeiter TO 'teamleiter' WITH GRANT OPTION;

REVOKE – Rechte entziehen

SQL-- Einzelnes Recht entziehen REVOKE INSERT ON mitarbeiter FROM 'anna'; -- Alle Rechte entziehen REVOKE ALL PRIVILEGES ON mitarbeiter FROM 'anna';

Rollen (Roles)

Rollen sind benannte Gruppen von Rechten. Statt jedem Benutzer einzeln Rechte zu vergeben, erstellst du eine Rolle mit den nötigen Rechten und weist die Rolle dem Benutzer zu. Bei 50 neuen Azubis änderst du nur die Rolle, nicht 50 einzelne Benutzer.

SQL (MySQL / PostgreSQL)-- Rolle erstellen CREATE ROLE azubi_rolle; -- Rechte der Rolle zuweisen GRANT SELECT, INSERT ON mitarbeiter TO azubi_rolle; -- Rolle einem Benutzer zuweisen GRANT azubi_rolle TO 'ben'; -- Rolle entziehen REVOKE azubi_rolle FROM 'ben';

Übersicht verfügbarer Rechte

RechtErlaubt
SELECTDaten lesen
INSERTDaten einfügen
UPDATEDaten ändern
DELETEDaten löschen
CREATETabellen erstellen
DROPTabellen löschen
ALL PRIVILEGESAlle oben genannten Rechte
Principle of Least Privilege

Jeder Benutzer bekommt nur die Rechte, die er wirklich braucht. Ein Webshop-Benutzer braucht SELECT, INSERT, UPDATE – aber kein DROP TABLE. Dieses Prinzip nennt sich "Principle of Least Privilege" (Prinzip der minimalen Rechte) und ist IHK-Pflichtwissen.

SQLite-Hinweis

SQLite hat kein Benutzer- und Rechtesystem. GRANT und REVOKE funktionieren dort nicht. Die Rechteverwaltung erfolgt auf Betriebssystemebene über Dateiberechtigungen.

Modul 10 · Sicherheit

SQL Injection & Best Practices

SQL Injection (SQLi) ist eine der häufigsten und gefährlichsten Sicherheitslücken in Webanwendungen. Seit Jahren auf Platz 1 der OWASP Top 10 (Open Web Application Security Project).

Was ist SQL Injection?

SQL Injection passiert wenn Benutzereingaben ungefiltert in einen SQL-Befehl eingebaut werden. Ein Angreifer schleust eigene SQL-Befehle ein und kann sie ausführen lassen.

Beispiel: Verwundbarer Login
PHP (unsicher)// Benutzereingabe direkt in SQL – NIEMALS so machen $sql = "SELECT * FROM users WHERE name='" . $_POST['name'] . "'"; // Normale Eingabe: name = 'ben' // Ergebnis: SELECT * FROM users WHERE name='ben' – ok // Angreifer gibt ein: name = ' OR '1'='1 // Ergebnis: SELECT * FROM users WHERE name='' OR '1'='1' // '1'='1' ist immer wahr – gibt ALLE Benutzer zurück!
Daten lesen

Datenleck

Alle Daten aus der Datenbank auslesen – Passwörter, Kreditkartendaten, Personaldaten.

Daten manipulieren

Manipulation

Einträge ändern oder löschen. Guthaben verändern, Noten fälschen.

Tabellen löschen

Zerstörung

DROP TABLE ausführen. Ohne Backup ist alles weg.

Schutz: Prepared Statements

Die wichtigste Gegenmaßnahme sind Prepared Statements (vorbereitete Anweisungen). SQL-Struktur und Daten werden getrennt übergeben. Benutzereingaben können niemals SQL-Befehle einschleusen.

PHP (sicher mit PDO)// SQL-Struktur und Daten getrennt $stmt = $pdo->prepare("SELECT * FROM users WHERE name = ?"); $stmt->execute([$_POST['name']]); // Egal was der Benutzer eingibt – es wird als reiner Datenwert behandelt // ' OR '1'='1 wird nicht als SQL interpretiert, sondern als Suchbegriff
Merkhilfe

Prepared Statements trennen den Bauplan (SQL) von den Zutaten (Daten). Der Bauplan steht fest, die Zutaten können keine Befehle enthalten.

Best Practices

1. Niemals DELETE oder UPDATE ohne WHERE

SQL-- FALSCH: löscht alle Zeilen DELETE FROM mitarbeiter; -- RICHTIG: erst mit SELECT prüfen, dann löschen SELECT * FROM mitarbeiter WHERE id = 5; DELETE FROM mitarbeiter WHERE id = 5;

2. Minimale Rechte (Principle of Least Privilege)

Jeder Benutzer bekommt nur die Rechte die er wirklich braucht. Kommt aus Modul 9 DCL – hier ist der Sicherheitsgrund dahinter.

3. Backups – 3-2-1-Regel

3 Kopien, auf 2 verschiedenen Medien, 1 davon extern. Ohne Backup sind DROP TABLE, versehentliches DELETE und Ransomware nicht behebbar.

4. Keine Fehlermeldungen nach außen

Datenbankfehlermeldungen im Browser geben Angreifern wertvolle Informationen über die Datenbankstruktur. Intern loggen, dem Benutzer nur eine generische Meldung zeigen.

5. Eingaben validieren

Auch mit Prepared Statements: Eingaben prüfen. Eine E-Mail-Adresse sollte wie eine E-Mail aussehen. Eine ID sollte eine Zahl sein. Je früher ungültige Eingaben abgefangen werden, desto besser.

Für die IHK

SQL Injection erklären können: Was ist es, wie funktioniert es, wie schützt man sich. Prepared Statements als Schutzmaßnahme kennen. Das Prinzip der minimalen Rechte benennen können.

Modul 11 · ER-Diagramme

ER-Diagramme – Grundlagen

Ein Entity-Relationship-Modell (ERM) ist ein grafisches Werkzeug zur Datenbankplanung. Du modellierst zuerst die reale Welt als Diagramm, bevor du eine einzige Zeile SQL schreibst.

Was ist ein Entity?

Eine Entität ist ein Objekt der realen Welt das gespeichert werden soll. Im ERM als Rechteck dargestellt, in SQL als Tabelle umgesetzt. Beispiele: Mitarbeiter, Abteilung, Produkt, Bestellung.

Was ist ein Attribut?

Ein Attribut ist eine Eigenschaft einer Entität. In Chen-Notation als Oval dargestellt, in SQL als Spalte umgesetzt. Beispiele bei Mitarbeiter: Name, E-Mail, Gehalt.

Was ist eine Beziehung?

Eine Beziehung (Relation) verbindet zwei Entitäten. In Chen-Notation als Raute dargestellt. Beispiel: Mitarbeiter "gehört zu" Abteilung.

Starke vs. schwache Entitäten

Starke Entität

Existiert eigenständig

Hat einen eigenen Primärschlüssel und kann ohne andere Tabellen existieren. Beispiele: Kunde, Fahrzeug, Artikel.

Schwache Entität

Existenzabhängig

Ohne die "Vater-Entität" macht sie keinen Sinn. Ihr PK enthält den FK der Vater-Entität. Beispiel: Kassenbon_Position ohne Kassenbon.

Beispiel schwache Entität

Eine Kassenbon-Position (Artikel X auf Bon Nr. 42) existiert nur, weil der Bon 42 existiert. Löschst du den Bon, entfallen auch alle Positionen. Der PK der Position enthält die bon_id als FK.

Modul 11 · ER-Diagramme

Kardinalitäten & Umsetzung in SQL

Kardinalitäten beschreiben wie viele Instanzen einer Entität mit wie vielen Instanzen einer anderen in Beziehung stehen können.

1:1 (Eins zu Eins)

Jeder Datensatz auf Seite A gehört zu genau einem Datensatz auf Seite B und umgekehrt.

Beispiel

Ein Mitarbeiter hat genau einen Ausweis. Ein Ausweis gehört zu genau einem Mitarbeiter.

SQL – 1:1 UmsetzungCREATE TABLE mitarbeiter_ausweis ( id INTEGER PRIMARY KEY, mitarb_id INTEGER UNIQUE NOT NULL, -- UNIQUE erzwingt 1:1 ausweisnr VARCHAR(20), FOREIGN KEY (mitarb_id) REFERENCES mitarbeiter(id) );

1:N (Eins zu Viele)

Ein Datensatz auf Seite A kann mit vielen Datensätzen auf Seite B verknüpft sein. Aber jeder Datensatz auf Seite B gehört zu genau einem auf Seite A.

Beispiel

Eine Abteilung hat viele Mitarbeiter. Jeder Mitarbeiter gehört zu genau einer Abteilung.

Regel: Der FK wandert immer auf die N-Seite (viele).

SQL – 1:N UmsetzungCREATE TABLE mitarbeiter ( id INTEGER PRIMARY KEY, name VARCHAR(100), abt_id INTEGER, -- FK auf der N-Seite FOREIGN KEY (abt_id) REFERENCES abteilung(id) );

M:N (Viele zu Viele)

Viele Datensätze auf Seite A können mit vielen Datensätzen auf Seite B verknüpft sein. Das kann nicht direkt mit FKs abgebildet werden – es braucht eine Zwischentabelle.

Beispiel

Ein Student besucht viele Kurse. Ein Kurs hat viele Studenten.

SQL – M:N Umsetzung mit ZwischentabelleCREATE TABLE student_kurs ( student_id INTEGER, kurs_id INTEGER, note DECIMAL(3,1), -- Beziehungsattribut möglich PRIMARY KEY (student_id, kurs_id), -- zusammengesetzter PK FOREIGN KEY (student_id) REFERENCES student(id), FOREIGN KEY (kurs_id) REFERENCES kurs(id) );

Modul 11 · ER-Diagramme

Erweitertes ERM

Das erweiterte ERM (EERM) ergänzt das klassische ERM um Konzepte wie Generalisierung und Spezialisierung.

IS-A Beziehung (Generalisierung)

IS-A beschreibt Vererbung. "Elektro_Auto IS-A Fahrzeug" bedeutet: jedes Elektro-Auto ist ein Fahrzeug und erbt alle Fahrzeug-Attribute. In SQL: der PK der Unterklasse ist gleichzeitig FK zur Oberklasse.

SQL – IS-A UmsetzungCREATE TABLE fahrzeug ( id INTEGER PRIMARY KEY, marke VARCHAR(50), baujahr INT ); CREATE TABLE elektro_auto ( fahrzeug_id INTEGER PRIMARY KEY, -- PK = FK zur Oberklasse akku_kwh DECIMAL(5,1), reichweite INT, FOREIGN KEY (fahrzeug_id) REFERENCES fahrzeug(id) );
Vererbung verstehen

Ein Elektro-Auto erbt Marke und Baujahr von Fahrzeug. In der Datenbank holst du dir diese Attribute per JOIN. Das ist das Datenbankmodell für objektorientierte Vererbung.

Vom ERM zur SQL-Tabelle – Vorgehen

Die IHK kann ein ERM-Diagramm geben und verlangen, dass du die SQL-Tabellen daraus ableitest. Vorgehensweise:

Schritt 1

Entitäten identifizieren

Jede Entität wird eine Tabelle. Name der Entität = Tabellenname.

Schritt 2

Attribute zuordnen

Attribute werden Spalten. Den Primärschlüssel festlegen.

Schritt 3

Beziehungen umsetzen

1:N → FK auf N-Seite. M:N → Zwischentabelle. 1:1 → FK mit UNIQUE.

Schritt 4

Reihenfolge beachten

Tabellen ohne FK zuerst erstellen (abteilung vor mitarbeiter).
